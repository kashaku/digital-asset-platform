## 1. 引言

### 1.1 编写目的

本《软件需求规格说明书》（Software Requirements Specification, SRS）旨在在系统实现与验收之前，对 AssetChain 产品的业务目标、功能边界、交互流程、异常策略及非功能指标进行形式化描述，为以下活动提供一致、可追踪的基准依据：

- **教学与答辩**：支撑课程设计、期末答辩与优秀项目展示中的需求可追溯性说明；
- **设计与实现**：为前端（React）、链上合约（Solidity / ERC-721）、存储（IPFS）及钱包集成（Ethers.js / MetaMask）提供统一需求基线；
- **测试与验收**：为功能测试、安全测试与演示脚本提供可验证的验收准则；
- **风险与约束管理**：明确 Web3 场景下特有的性能、成本与安全约束，降低实现偏差。

本文档的预期读者包括：课程指导教师、项目组开发与测试人员、评审专家及潜在用户代表（创作者与收藏者）。

### 1.2 项目背景与痛点分析

#### 1.2.1 项目背景

数字资产（图像、音频、三维模型、文档等）在互联网经济中的创作、传播与二次流通日益频繁。传统中心化数字资产平台通常依赖单一运营主体完成托管、展示、撮合与结算。AssetChain 拟采用 **公链账户体系 + 智能合约业务规则 + IPFS 内容寻址** 的组合架构，在 **Polygon** 网络上部署符合 **ERC-721** 标准的非同质化通证（NFT），将“权利凭证”与“媒体内容”在逻辑上解耦：链上记录不可篡改的权属与交易规则，链下（IPFS）承载大体积数字内容，从而在工程上兼顾可验证性与可扩展性。

#### 1.2.2 传统中心化模式的主要痛点

| 维度 | 传统中心化平台典型问题 | AssetChain 的应对思路（需求层面） |
|------|------------------------|-----------------------------------|
| **佣金与规则** | 平台抽成比例高、规则单方变更 | 版税与分账逻辑由合约编码，关键参数上链可审计（在合规前提下） |
| **封禁与可用性** | 账号或内容因平台政策被限制，资产“可见性”依赖平台 | 通证与合约在公链上可独立验证；前端仅为访问入口之一 |
| **创作者二次收益** | 二级市场收益分配依赖平台结算，透明度不足 | 二级市场成交时由合约自动划转版税至创作者地址 |
| **确权与时效** | 确权依赖平台数据库记录，争议时举证成本高 | 链上铸造与转移事件可公开验证；结合时间轴展示所有权变更路径 |
| **溯源** | 历史流转记录分散在平台日志中，跨平台不可组合 | 以链上事件与交易为核心数据源构建溯源时间轴 |

上述对比用于说明产品动机；具体能力以本文档 **第 3 章功能需求** 与 **第 4 章非功能需求** 为准。

### 1.3 核心专业术语定义

| 术语 | 定义 |
|------|------|
| **Web3** | 以用户自持密钥、链上状态与开放协议为核心的应用范式；本项目中体现为钱包登录、链上读写与可验证数据。 |
| **智能合约（Smart Contract）** | 部署在区块链上的确定性程序代码，由交易触发执行；本项目业务规则（铸造、上架、成交、版税）以 Solidity 实现。 |
| **ERC-721** | 以太坊生态中非同质化通证（NFT）的标准接口与事件约定；每个 Token 具有唯一 `tokenId`。 |
| **Polygon** | 兼容以太坊虚拟机的二层/侧链生态之一；本系统合约部署目标网络，以降低部分场景下的交易成本与确认时间（相对主网而言，仍以实际网络状况为准）。 |
| **Gas / Gas Fee** | 为执行链上交易或合约调用所支付的计算与存储资源费用；费用由网络规则与交易复杂度决定。 |
| **钱包（Wallet）** | 管理私钥与链上地址的软件或硬件；本项目以 **MetaMask** 浏览器插件为交互载体。 |
| **Ethers.js** | 以太坊兼容链的 JavaScript 库，用于前端与节点 RPC 交互、合约 ABI 调用与事件监听。 |
| **IPFS（星际文件系统）** | 基于内容寻址的分布式文件系统；文件上传后得到 **CID**（内容标识符），用于定位与校验内容。 |
| **CID（Content Identifier）** | IPFS 中标识内容对象的字符串；内容变更则 CID 变更，可用于完整性关联。 |
| **Minting（铸造）** | 调用合约创建新的 ERC-721 Token，并与元数据（通常含 `tokenURI` 指向元数据资源）建立绑定关系。 |
| **版税（Royalty）** | 在二级市场成交时，从成交额中按约定比例向创作者或受益人划转的收益分配机制；具体实现依赖合约与市场撮合逻辑。 |
| **Offer（出价）** | 买方在链上或链下签名/合约托管等方式表达购买意愿与价格；本项目需求包含“一口价”与“出价”两种模式，具体链上模型以实现阶段选定的合约为准。 |
| **确权** | 在本项目中指：对特定数字内容与链上 Token 的对应关系、当前持有者、合约地址与 `tokenId` 等信息的可验证展示。 |
| **溯源** | 基于链上转移、交易、拍卖等事件构建的所有权变更时间序列。 |

## 2. 总体描述

### 2.1 产品受众类型

| 受众 | 主要目标 | 典型使用场景 |
|------|----------|----------------|
| **数字创作者** | 铸造资产、设定版税、获得二级市场分成 | 上传作品、填写元数据、发起铸造交易、管理已发行资产 |
| **收藏者与投资者** | 发现资产、完成购买或出价、管理持仓 | 浏览市场、筛选资产、发起购买/Offer、查看持仓与历史 |
| **课程评审与指导教师** | 理解系统边界与技术路线 | 阅读 SRS、对照演示验证功能与安全性说明 |
| **技术人员（组内）** | 实现与联调 | 依据接口与异常策略实现前后端与合约 |

系统不假设终端用户具备 Solidity 开发能力，但应假设其具备 **安装 MetaMask、理解交易需签名与消耗 Gas、能保存助记词/私钥安全** 的基础认知；界面需提供必要引导以降低误操作率。

### 2.2 运行环境与技术选型约束

#### 2.2.1 客户端运行环境

- **浏览器**：推荐使用最新稳定版的 Chromium 系或 Firefox（具体最低版本可在实现阶段锁定，答辩环境需提前验证）。
- **插件**：MetaMask（或与 Ethers.js 兼容的注入式 `window.ethereum` 提供者）。
- **网络**：可访问 Polygon RPC 节点、IPFS 网关或自定义节点；需考虑校园网等环境对 RPC 域名的访问策略。

#### 2.2.2 技术栈约束（与实现一致）

| 层次 | 技术选型 | 约束说明 |
|------|----------|----------|
| 前端应用层 | React + TailwindCSS | UI 组件与样式需保持可维护性；链上状态与本地 UI 状态需清晰分离。 |
| 链交互与认证 | Ethers.js + MetaMask | 所有写链操作须经用户签名授权；需处理账户切换、链切换、RPC 失败等。 |
| 智能合约 | Solidity，部署于 Polygon | 遵循 ERC-721；关键函数需可重入防护与权限控制（见第 4.1 节）。 |
| 分布式存储 | IPFS | 大文件不经由链上存储；元数据与媒体文件的 CID 需与 `tokenURI` 策略一致。 |

#### 2.2.3 假设与依赖

- 依赖第三方 **RPC 服务商** 与 **IPFS 节点/网关** 的可用性；当服务不可用时，系统应降级提示而非静默失败。
- 链上最终性以网络共识为准；“秒级确权”指 **在交易被网络接受并确认后的可验证状态**，不包含用户网络延迟或 RPC 排队等不可控因素；界面需对确认过程做明确反馈（见第 4.3 节）。

## 3. 功能需求分析

本章对每个模块给出 **前置条件**、**操作流程**、**异常处理** 及关键验收要点。流程中的“系统”指 AssetChain 前后端与合约的集合。

### 3.1 钱包登录与授权认证

#### 3.1.1 需求概述

用户通过 MetaMask 与站点建立会话：读取当前链上地址作为身份标识；对写链操作进行逐笔授权。系统不托管用户私钥。

#### 3.1.2 前置条件

- 用户浏览器已安装 MetaMask 且已完成基础初始化（至少存在一个可用账户）。
- 用户允许站点访问 `window.ethereum`（首次连接需授权）。
- RPC 可连通 Polygon 目标网络（或系统提示用户切换至正确 `chainId`）。

#### 3.1.3 操作流程

1. 用户进入站点，点击“连接钱包”。
2. 系统调用 `eth_requestAccounts`（或等价流程）获取当前账户地址并展示 shortened 地址。
3. 系统检测当前网络 `chainId`；若与 Polygon 目标不一致，提示用户切换网络并提供一键切换（若钱包支持 `wallet_switchEthereumChain`）。
4. 用户进行铸造、购买、出价等写链操作前，系统组装交易/合约调用参数，唤起 MetaMask 签名；用户确认后广播交易。
5. 系统订阅交易回执或轮询 `tx.hash` 状态，在确认后更新 UI 与会话内状态。

#### 3.1.4 异常处理

| 异常类型 | 系统行为 |
|----------|----------|
| 未安装钱包 | 展示安装引导与文档链接；禁用写链按钮。 |
| 用户拒绝连接 | 保留只读模式（若产品允许）；提示可重试。 |
| 链错误 | 阻断写链操作；展示当前链与目标链说明。 |
| 用户拒绝签名 | 不广播交易；保留表单数据；提示“已取消”。 |
| RPC 超时/限流 | 重试策略（退避）；提示更换 RPC 或稍后重试。 |
| 账户切换（`accountsChanged`） | 刷新会话上下文；若与订单/表单相关则提示用户核对地址。 |

#### 3.1.5 验收要点

- 未连接钱包时，敏感操作不可执行且有明确提示。
- 连接后地址展示与链 ID 校验符合配置。
- 拒绝签名与 RPC 失败均有可理解的错误文案（可对常见错误码做映射）。

### 3.2 数字资产铸造模块

#### 3.2.1 需求概述

用户将数字文件上传至 IPFS 获得 CID，填写资产元数据（名称、描述、属性等），设定版税比例（在合约允许范围内），调用 ERC-721 合约完成铸造；链上生成唯一 `tokenId` 并与元数据 URI 关联。

#### 3.2.2 前置条件

- 用户已完成钱包连接且网络正确。
- IPFS 上传端点可用；文件大小与类型符合实现阶段定义的上限与白名单（需在用户上传前明示）。
- 用户账户具有足够 **原生代币**（如 POL/MATIC，以当时网络为准）支付 Gas。

#### 3.2.3 操作流程

1. 用户选择本地文件；系统进行前端校验（类型、大小）。
2. 系统将文件上传至 IPFS，获得 **媒体 CID**；可选：生成缩略图并同样上传得第二 CID。
3. 用户填写元数据字段；系统生成符合 **OpenSea 兼容习惯**（若采用）的 JSON 元数据，上传至 IPFS 得 **元数据 CID**。
4. 用户设定版税比例（若由合约参数表达）；系统展示预估 Gas（若可估算）与风险提示。
5. 用户确认后，系统调用合约 `mint`（或等价函数），传入 `tokenURI`（通常为指向元数据 JSON 的 IPFS URL）及版税相关参数。
6. 交易确认后，系统展示 **TokenID、合约地址、交易哈希、IPFS CID**，并提供跳转区块浏览器与网关预览链接。

#### 3.2.4 异常处理

| 异常类型 | 系统行为 |
|----------|----------|
| 上传 IPFS 失败 | 中止铸造；保留表单；提示重试或检查网络。 |
| 元数据不合规 | 前端校验拦截；给出字段级错误。 |
| Gas 不足 | 在签名前若可检测则提示充值；否则展示链上错误映射。 |
| 合约 `revert` | 展示可读错误；记录 `tokenId` 未生成状态。 |
| 重复提交 | 通过按钮 loading 与禁用防止双花式重复广播；必要时以 nonce 与 UI 状态机约束。 |

#### 3.2.5 验收要点

- 铸造成功后可在区块浏览器验证 Token 与 `tokenURI`。
- 元数据 JSON 可被公共网关解析（在网关可达前提下）。
- 版税参数与合约存储一致（可通过只读调用核对）。

### 3.3 去中心化交易模块

#### 3.3.1 需求概述

提供数字资产的市场列表：支持浏览、筛选（如按价格区间、创作者、合约集合等，以实现为准）；支持 **一口价购买** 与 **出价（Offer）** 两种模式。成交与资金流转由智能合约保证，前端负责展示与触发交易。

#### 3.3.2 前置条件

- 买家/卖家钱包已连接且网络正确。
- 资产已铸造且当前所有者地址有效。
- **购买**：买家具备足够支付对价的代币（若使用原生币或 ERC-20，以实现阶段为准）及 Gas。
- **上架/接受出价**：卖家对合约具有授权（如 `setApprovalForAll` 或单 Token 授权），且授权未过期或被撤销。

#### 3.3.3 操作流程（浏览与筛选）

1. 用户进入市场页，系统从链上索引服务或直连事件日志/合约视图拉取列表数据（性能方案以实现为准）。
2. 用户应用筛选条件，系统更新列表排序与分页。
3. 用户进入资产详情页，查看价格、卖家、版税说明、历史记录入口。

#### 3.3.4 操作流程（一口价购买）

1. 卖家以合约规定方式上架（固定价格、库存为 1 的 NFT 场景）。
2. 买家点击购买，系统校验授权与余额，唤起付款 + Gas 签名。
3. 交易确认后，所有权转移；版税从成交额中按合约规则自动分配（见 3.3.6）。

#### 3.3.5 操作流程（出价 Offer）

1. 买家输入出价金额与有效期（若合约支持），提交后按设计完成：链上挂单或链下签名后由卖家接受等（**具体状态机以实现阶段合约为准**）。
2. 卖家可在资产详情页查看 Offer 列表并接受合适出价。
3. 成交后生成与一口价类似的所有权变更与分账结果。

#### 3.3.6 自动化版税（与交易耦合）

- 在二级市场成交路径中，合约须按资产铸造时设定的版税比例，将对应金额划转至创作者受益地址。
- 前端须在详情页展示 **版税比例与受益方地址（或别名）** 的只读信息，避免用户误解到手金额。

#### 3.3.7 异常处理

| 异常类型 | 系统行为 |
|----------|----------|
| 未授权或授权不足 | 引导用户发起 `approve` / `setApprovalForAll`；说明一次性授权与最小权限原则。 |
| 余额不足 | 购买前提示；链上失败时解析 `revert` 原因。 |
| 资产已售出或下架 | 列表与详情实时或准实时更新；禁用购买按钮。 |
| Offer 过期 | 自动标记失效；不接受成交。 |
| 索引延迟 | UI 标注“链上确认可能存在延迟”；提供刷新与交易哈希核对入口。 |

#### 3.3.8 验收要点

- 典型路径：上架 → 购买 → 所有权变更 → 版税分账可在链上验证。
- Offer 路径：创建 → 接受/拒绝/过期 状态与合约一致。

### 3.4 资产溯源与个人中心模块

#### 3.4.1 需求概述

**确权与溯源**：对任意已铸造资产，展示链上确权信息（合约地址、`tokenId`、当前持有者、铸造交易、元数据链接）；以 **时间轴** 形式聚合 `Transfer` 及相关市场合约事件，展示历史所有权变更路径。

**个人中心**：展示当前地址持有的 NFT、发起的交易、收到的版税记录（若可从事件解析）、以及铸造过的作品列表。

#### 3.4.2 前置条件

- 用户已连接钱包（查看“我的资产”）；查看他人资产仅需合约与 `tokenId` 或链接。
- 区块浏览器或内部索引可查询历史日志（在仅 RPC 情况下可能较慢，需 UX 补偿）。

#### 3.4.3 操作流程（溯源）

1. 用户在详情页打开“溯源/历史”页签。
2. 系统自创世 `mint` 起向后拉取 `Transfer(from,to,tokenId)` 事件，按区块时间排序。
3. 若存在市场合约地址，合并解析拍卖成交、接受出价等事件（以实现为准），映射到时间轴节点。
4. 每一节点展示：时间（区块时间戳）、交易哈希、转出/转入地址、简短说明。

#### 3.4.4 操作流程（个人中心）

1. 用户进入“个人中心”，系统以当前 `account` 为键查询持仓（可通过 ERC-721 `balanceOf` + `tokenOfOwnerByIndex` 或枚举方案，以实现为准）。
2. 展示列表卡片：缩略图、名称、`tokenId`、当前是否上架。
3. 提供跳转至铸造、上架、查看链上记录入口。

#### 3.4.5 异常处理

| 异常类型 | 系统行为 |
|----------|----------|
| 元数据网关不可达 | 展示占位图与 CID；提示更换网关或稍后重试。 |
| 历史事件过多 | 分页或“加载更多”；标注查询范围。 |
| 非本合约资产链接 | 校验 `chainId` 与合约地址白名单；防止钓鱼合约误展示为官方。 |

#### 3.4.6 验收要点

- 时间轴顺序与链上 `Transfer` 一致（在相同 RPC 数据源下）。
- 个人中心持仓与链上 `ownerOf` 一致。

## 4. 非功能需求

### 4.1 安全性需求（结合 Web3 特性）

| 编号 | 需求描述 | 说明与建议实现方向 |
|------|----------|--------------------|
| SEC-01 | **重入防护** | 涉及外部调用的合约函数遵循 **checks-effects-interactions**；对可重入路径使用 ReentrancyGuard 或等价模式。 |
| SEC-02 | **权限最小化** | 仅对市场合约授予必要 NFT 授权；文档与用户引导中明确授权范围与撤销方式。 |
| SEC-03 | **整数溢出** | 使用 Solidity 0.8+ 内置检查或 SafeMath（低版本）；版税与价格计算避免截断与精度问题。 |
| SEC-04 | **前端供应链安全** | 锁定依赖版本；构建链路校验；避免将私钥或助记词写入前端代码。 |
| SEC-05 | **钓鱼与假冒合约** | 前端配置可信合约地址列表；网络切换后重新校验；对外分享链接包含 `chainId` 与合约地址。 |
| SEC-06 | **多维度风险扫描** | 在里程碑节点使用 Slither、Mythril 等静态分析工具对合约扫描；对高危项建立整改清单（教学项目亦应形成记录）。 |
| SEC-07 | **输入校验** | 对 URI、版税比例、价格等参数在合约与前端双重校验，防止恶意或越界输入。 |

### 4.2 性能需求（Gas 与 IPFS）

| 编号 | 需求描述 | 指标或策略 |
|------|----------|------------|
| PERF-01 | **Gas 优化** | 合并存储槽、避免冗余写入、谨慎使用循环遍历无界数组；复杂列表链下索引、链上仅存根或事件。 |
| PERF-02 | **批量操作** | 若教学范围需要，评估批量上架/撤单等的 Gas 与失败回滚成本，非必需则不强制。 |
| PERF-03 | **IPFS 检索** | 支持可配置网关列表与失败重试；对大文件采用分块与进度显示；缓存元数据 JSON 的解析结果（短期、可失效）。 |
| PERF-04 | **RPC 调用** | 合并只读调用（multicall）；列表页使用分页；避免详情页重复请求相同数据。 |

### 4.3 可用性需求（链上延迟的 UI/UX 补偿）

| 编号 | 需求描述 |
|------|----------|
| UX-01 | 所有写链操作展示 **pending** 状态：交易已广播、等待确认、已确认三阶段文案与可视化进度。 |
| UX-02 | 提供 **交易哈希** 可复制与跳转区块浏览器，便于用户自助核对。 |
| UX-03 | 对 Gas 价格波动，在可估算场景展示 **费用区间或“仅供参考”** 声明，避免误导。 |
| UX-04 | 账户切换、网络切换时，全局提示并刷新依赖地址的页面数据，防止“所见非所签”。 |
| UX-05 | 关键不可逆操作（购买、接受出价、授权）使用二次确认模态框，列出对手方、价格、版税与预估总支出。 |
